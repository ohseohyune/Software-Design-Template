| 역할 | 누구인가 | 원하는 것 | 두려워하는 것 |
|-----|--------|---------|------------|
| 최종 사용자 |   |        |            |

| 운영자 (배포/유지보수) | | | |
| 이해관계자 | | | |

왜 설계에서 이걸 정리하는가?
설계 결정을 내릴 때 "누구 입장에서 좋은 설계인가" 를 명확히 하기 위해서다.
이해관계자가 다르면 요구사항이 충돌한다:

최종 사용자: "버튼 누르면 바로 반응해야 해" → 낮은 latency
운영자: "서버 비용 줄여야 해" → 처리 배치화, 낮은 throughput
보안팀: "모든 요청 로깅해야 해" → 저장 비용 증가

이 충돌을 모르고 설계하면 나중에 구조를 갈아엎어야 한다.

-> 혼자 하는 토이 프로젝트라면 이 칸은 그냥 스킵해도 된다. 이 항목이 진짜 중요해지는 건 팀 프로젝트나 실제 배포되는 시스템일 때

# 설계 검토 체크리스트 (Design Review)

> 설계를 마치기 전 최종 점검.

## 완전성 체크
- [ ] 모든 FR이 어떤 컴포넌트에서 구현되는지 추적 가능한가
- [ ] 모든 컴포넌트의 책임이 명확하게 정의되었는가
- [ ] 모든 외부 의존성이 식별되었는가
- [ ] 데이터 흐름이 끊기지 않고 소스에서 싱크까지 이어지는가

## 일관성 체크
- [ ] 여러 곳에서 같은 개념을 다른 이름으로 부르지 않는가
- [ ] 인터페이스 타입이 컴포넌트 간에 일치하는가
- [ ] 레이어 의존성 규칙이 지켜지는가

## 리스크 체크
- [ ] 가장 불확실한 부분이 식별되고 대응 방안이 있는가
- [ ] 핵심 실패 시나리오에 복구 전략이 있는가
- [ ] 성능 병목이 될 가능성이 있는 곳이 식별되었는가

## Trade-off 명시
- [ ] 주요 아키텍처 결정의 trade-off가 ADR에 기록되었는가
- [ ] 포기한 대안들이 기록되었는가 (나중에 같은 고민 반복 방지)
